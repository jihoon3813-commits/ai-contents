# 안티그래비티 단계별 개발 프롬프트

## 멀티플랫폼 AI 콘텐츠 제작 SaaS

---

# 공통 프로젝트 규칙

아래 공통 규칙은 모든 Phase에서 반드시 유지한다.

```text
이 프로젝트는 사용자가 브랜드, 콘텐츠 목적, 플랫폼, 카테고리, 주제, 키워드, 글자 수, 이미지 수, 문체와 실제 경험을 입력하면 플랫폼별로 최적화된 콘텐츠를 생성하는 멀티플랫폼 AI 콘텐츠 제작 SaaS다.

지원 플랫폼:
- 워드프레스
- Blogger
- 티스토리
- 네이버 블로그
- 인스타그램

기술 스택:
- Next.js 최신 안정 버전
- App Router
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase PostgreSQL
- Supabase Auth
- Supabase Storage
- React Hook Form
- Zod
- TanStack Query
- TipTap
- Playwright

공통 개발 원칙:

1. 작업 전에 현재 코드 구조와 데이터베이스 상태를 먼저 분석한다.
2. 현재 Phase 범위 밖의 기능은 임의로 구현하지 않는다.
3. 기존에 정상 작동하는 기능을 삭제하거나 변경하지 않는다.
4. 모든 사용자 입력은 Zod로 검증한다.
5. 모든 외부 API 키와 인증정보는 서버 환경변수로 관리한다.
6. Supabase service role key를 클라이언트에 노출하지 않는다.
7. 모든 사용자 데이터에는 workspace_id를 사용한다.
8. 핵심 테이블에 Supabase Row Level Security를 적용한다.
9. 사용자는 자신이 속한 워크스페이스 데이터만 접근할 수 있다.
10. 서버 오류와 사용자 입력 오류를 분리해서 처리한다.
11. 사용자에게는 이해하기 쉬운 한국어 오류 메시지를 표시한다.
12. 모든 비동기 작업은 로딩·성공·실패 상태를 표시한다.
13. 생성형 AI 결과는 JSON 스키마로 검증한 뒤 저장한다.
14. AI가 사용자가 제공하지 않은 경험, 가격, 통계, 인증을 사실처럼 생성하지 않도록 한다.
15. 생성 콘텐츠의 기본 상태는 NEEDS_REVIEW로 저장한다.
16. 외부 플랫폼 발행은 기본적으로 초안 상태로 처리한다.
17. 모든 삭제는 우선 소프트 삭제를 사용한다.
18. 민감 정보와 토큰은 로그에 기록하지 않는다.
19. TypeScript strict 모드를 유지한다.
20. 구현 후 lint, typecheck, unit test, integration test, Playwright E2E 테스트를 실행한다.
21. 테스트 실패가 있으면 원인을 수정한 뒤 다시 실행한다.
22. 실제 브라우저에서 핵심 사용자 흐름을 검증한다.
23. 작업 완료 후 다음 내용을 보고한다.
   - 구현 내용
   - 생성·수정 파일
   - 데이터베이스 변경
   - 테스트 결과
   - 실제 브라우저 검증 결과
   - 알려진 문제
   - 다음 Phase 진행 시 주의사항
```

# Phase 1 프롬프트

## 프로젝트 기반, 인증, 워크스페이스

```text
현재 프로젝트의 Phase 1을 구현한다.

이번 Phase의 목표는 프로젝트 기본 구조, 인증, 사용자 프로필, 워크스페이스와 권한 분리를 완성하는 것이다.

현재 Phase 범위:

1. Next.js App Router 프로젝트 기본 구조
2. TypeScript strict 설정
3. Tailwind CSS
4. shadcn/ui
5. Supabase 클라이언트와 서버 클라이언트 구성

profiles:

workspaces:

- updated_at

3. 기본 워크스페이스 생성
4. workspace_members에 OWNER 등록
5. FREE subscriptions 생성
6. 모든 작업이 하나의 트랜잭션 또는 안전한 트리거 흐름으로 동작해야 한다.

화면 요구사항:

공개 페이지:
- /
- /login
- /signup
- /forgot-password
- /reset-password

보호 페이지:
- /dashboard
- /settings/profile
- /settings/workspace

대시보드 기본 레이아웃:
- 좌측 사이드바
- 상단 헤더
- 모바일 메뉴
- 사용자 이름 표시
- 워크스페이스 이름 표시
- 로그아웃 버튼

인증 요구사항:

- 이메일 형식 검증
- 비밀번호 최소 8자
- 비밀번호 확인 일치
- 중복 이메일 오류 처리
- 로그인 실패 메시지
- 비밀번호 재설정 이메일 발송
- 세션 만료 처리
- 로그인 후 원래 접근 페이지로 복귀
- 새로고침 후 로그인 상태 유지

RLS 요구사항:

- 사용자는 자신의 profile만 조회·수정 가능
- 사용자는 자신이 멤버인 workspace만 조회 가능
- workspace OWNER만 workspace 기본정보 수정 가능
- 사용자는 자신이 소속된 workspace_members만 조회 가능
- 다른 워크스페이스 데이터 접근 차단

UI 요구사항:

- 한국어 UI
- 반응형
- 폼 제출 중 버튼 비활성화
- 오류 메시지를 입력 필드 아래 표시
- 성공·실패 토스트
- 접근성 label 적용
- 키보드 탐색 가능

테스트 요구사항:

단위 테스트:
- 회원가입 입력 스키마
- 로그인 입력 스키마
- 워크스페이스 slug 생성
- 권한 확인 함수

통합 테스트:
- 신규 가입 시 profile, workspace, member, subscription 생성
- 다른 사용자 워크스페이스 조회 차단
- 비밀번호 재설정 흐름

Playwright E2E:
1. 이메일 회원가입
2. 로그인
3. 대시보드 접근
4. 새로고침 후 세션 유지
5. 로그아웃
6. 보호 페이지 접근 차단
7. Google 로그인 버튼 렌더링 확인

구현 전 먼저 다음을 제시한다.

1. 현재 프로젝트 상태 분석
2. 추천 폴더 구조
3. 설치할 패키지
4. 필요한 환경변수
5. 데이터베이스 마이그레이션 계획
6. RLS 정책 계획
7. 구현 파일 목록
8. 테스트 계획

제외 범위:

- 브랜드 관리
- 콘텐츠 생성
- AI API 연동
- 플랫폼 외부 연동
- 콘텐츠 에디터
- SEO 점수
- 결제 시스템

완료 조건:

- 신규 가입이 정상 작동한다.
- profile, workspace, workspace_member, subscription이 자동 생성된다.
- 사용자는 자신의 워크스페이스만 조회할 수 있다.
- 보호된 페이지가 정상 작동한다.
- 로그인 상태가 유지된다.
- 로그아웃이 정상 작동한다.
- 모바일과 데스크톱 레이아웃이 정상이다.
- lint, typecheck, 테스트가 모두 통과한다.
```

# Phase 2 프롬프트

## 브랜드 및 브랜드 보이스 관리

```text
현재 프로젝트의 Phase 2를 구현한다.

Phase 1의 인증, 워크스페이스, RLS와 대시보드 기능은 유지한다.
기존 정상 기능을 변경하거나 삭제하지 않는다.

이번 Phase의 목표는 사용자가 브랜드를 만들고, 브랜드 정보와 문체를 관리하며, 과거 작성글을 기반으로 브랜드 보이스를 분석할 수 있게 하는 것이다.

현재 Phase 범위:

1. 브랜드 목록
2. 브랜드 생성
3. 브랜드 상세 조회
4. 브랜드 수정
5. 브랜드 복제
6. 브랜드 소프트 삭제
7. 기본 브랜드 설정
8. 브랜드 보이스 설정
9. 브랜드 샘플 텍스트 등록
10. 샘플 텍스트 분석 상태 관리
11. 브랜드 보이스 분석용 서비스 인터페이스
12. 실제 AI 호출 전 Mock 분석 기능
13. 온보딩 브랜드 생성 화면

데이터베이스 테이블:

brands:
- id
- workspace_id
- name
- industry
- description
- tagline
- website_url
- target_audience
- customer_problems
- products_services
- core_values
- default_cta
- legal_notice
- is_default
- created_by
- updated_by
- created_at
- updated_at
- deleted_at

brand_voice_profiles:
- id
- brand_id, unique
- formal_level, 1~5
- sentence_length, 1~5
- expertise_level, 1~5
- emotional_level, 1~5
- sales_level, 1~5
- humor_level, 1~5
- emoji_level, 1~5
- question_level, 1~5
- honorific_style
- preferred_phrases
- forbidden_phrases
- proprietary_terms
- intro_style
- closing_style
- cta_style
- analysis_summary
- approved_at
- created_at
- updated_at

brand_samples:
- id
- workspace_id
- brand_id
- title
- source_type: PASTED, FILE
- source_url
- raw_text
- file_path
- character_count
- analysis_status: PENDING, PROCESSING, COMPLETED, FAILED
- analysis_result
- created_by
- created_at

화면:

- /brands
- /brands/new
- /brands/[id]
- /brands/[id]/edit
- /brands/[id]/voice

- /onboarding/brand

브랜드 생성 폼:

필수:
- 브랜드명
- 업종

선택:
- 한 줄 소개
- 상세 설명
- 홈페이지
- 주요 고객
- 고객 문제
- 주요 상품·서비스
- 핵심 가치
- 기본 CTA
- 법적 고지 문구

브랜드 보이스 설정:

슬라이더:
- 공식성
- 문장 길이
- 전문성
- 감정 표현
- 판매 성향
- 유머

추가 설정:
- 존댓말 스타일
- 선호 표현
- 금지 표현
- 브랜드 고유 용어

각 슬라이더는 1~5 정수로 저장한다.
태그형 입력은 중복을 방지한다.

브랜드 샘플 등록:

지원 방식:
- 텍스트 붙여넣기
- TXT 파일 업로드
- DOCX 파일 업로드

검증:
- 빈 파일 금지
- 파일 크기 제한

- 최대 글자 수
- UTF-8 텍스트 정규화

MVP Mock 브랜드 보이스 분석:

실제 AI 연동 전 다음을 코드 기반으로 분석한다.

- 총 글자 수
- 평균 문장 길이
- 평균 문단 길이
- 존댓말 사용 비율
- 질문형 문장 비율
- 느낌표 사용 빈도
- 이모지 빈도
- 자주 사용하는 어미
- 자주 사용하는 접속사
- 반복되는 표현

분석 결과를 brand_samples.analysis_result에 저장한다.
여러 샘플의 결과를 합산하여 brand_voice_profiles.analysis_summary에 반영할 수 있어야 한다.

기능 요구사항:

- 워크스페이스별 데이터 완전 분리
- 기본 브랜드는 워크스페이스당 하나만 허용
- 새로운 기본 브랜드 지정 시 기존 기본값 자동 해제
- 브랜드 삭제 시 연결 데이터가 있으면 경고
- 삭제는 deleted_at을 이용한 소프트 삭제
- 삭제된 브랜드는 기본 목록에서 제외
- 브랜드 복제 시 보이스 설정도 복제
- 샘플 원문과 분석 결과를 구분해서 저장
- 분석 결과는 사용자가 수정하고 승인할 수 있어야 함

대시보드 변경:

- 좌측 메뉴에 브랜드 추가
- 브랜드 수 표시
- 기본 브랜드 표시
- 브랜드가 없으면 생성 유도 카드 표시

API 또는 Server Action:

- createBrand
- updateBrand
- deleteBrand
- duplicateBrand
- setDefaultBrand
- createBrandSample
- deleteBrandSample
- analyzeBrandSamples
- updateBrandVoice
- approveBrandVoice

- 다른 워크스페이스 브랜드 접근 차단

테스트:

단위 테스트:
- 브랜드 폼 스키마
- 슬라이더 범위
- 기본 브랜드 전환
- 태그 중복 제거
- Mock 문체 분석

통합 테스트:
- 브랜드 생성·수정·삭제
- 기본 브랜드 변경
- 샘플 등록
- 샘플 분석 결과 저장
- 브랜드 복제
- 권한별 수정 제한

Playwright E2E:
1. 로그인
2. 브랜드 생성
3. 브랜드 보이스 설정
4. 샘플 텍스트 붙여넣기
5. 분석 실행
6. 결과 수정
7. 보이스 승인
8. 브랜드 목록 확인

구현 전 다음을 제시한다.

1. 기존 Phase 1 구조 분석
2. DB 마이그레이션 계획
3. 화면 구조
4. 컴포넌트 구조
5. Server Action 또는 API 구조
6. 파일 업로드 처리 계획
7. 테스트 계획

제외 범위:

- 실제 AI API 호출
- 콘텐츠 프로젝트 생성
- 플랫폼별 콘텐츠
```

# 안티그래비티가 범위를 벗어날 때 사용하는 제어 프롬프트

```text
현재 작업 범위를 다시 제한한다.

이번 작업에서는 현재 Phase 요구사항에 명시된 기능만 구현한다.

다음 행동을 하지 않는다.

- 다음 Phase 기능 선행 구현
- 기존 테이블 임의 삭제
- 기존 API 경로 변경
- 현재 정상 기능의 전면 리팩터링
- 기술 스택 임의 변경
- 인증 방식 변경
- 데이터베이스 제공자 변경
- UI 라이브러리 변경
- 테스트 생략
- 임시 하드코딩
- 비밀키 코드 삽입
- Mock 데이터를 운영 코드에 고정
- 요구사항에 없는 자동 발행 구현

필요한 구조 확장이 있으면 직접 구현하기 전에 다음 내용을 먼저 보고한다.

1. 확장이 필요한 이유
2. 변경될 파일
3. DB 영향
4. 기존 기능 영향
5. 대안
6. 권장안

현재 Phase의 완료 조건을 우선 충족한다.
```
